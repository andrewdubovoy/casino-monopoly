# Feature-Sliced Design  структура для Казино Монополия

Рекомендованная корневая структура (в `src/`):

- `app/`  точка входа приложения, роутинг, провайдеры (i18n, store)
- `pages/`  страницы (страницы собирают сущности и фичи)
- `processes/`  сложные процессы или сценарии (мульти-стадийные flows)
- `widgets/`  переиспользуемые UI-компоненты, агрегирующие фичи
- `features/`  бизнес-функции (каждая фича инкапсулирует логику/компоненты)
- `entities/`  доменные сущности (игрок, банк, игровое поле, ставка)
- `shared/`  общие утилиты, UI-библиотека, стили, api-клиенты

Пример дерева для одной фичи `game-play`:

- `src/features/game-play/ui/`  визуальные компоненты фичи
- `src/features/game-play/lib/`  утилиты и бизнес-логика
- `src/features/game-play/api/`  обращения к backend (сервисные слои)
- `src/features/game-play/model/`  локальный state (hooks, slices)

Конвенции и правила:
- Файлы именовать описательно: `StartGameButton.tsx`, `useGameState.ts`.
- Срезы (slices) должны быть максимально изолированы и иметь четкий API.
- `entities` содержит pure-модели (типизация, трансформеры, валидаторы).
- `shared`  только действительно общие вещи, не feature-specific.
- Тесты располагаются рядом с реализацией: `component.test.tsx`.

Примечание: начните с малого  создайте `app/`, `pages/` и `entities/player`.
