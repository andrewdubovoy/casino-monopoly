## Промпт для обновления знаний проекта (используйте при общении с новой моделью)

Инструкция: вставьте этот промпт в начало сессии новой модели, чтобы она получила контекст проекта.

---

Проект: Казино Монополия

Краткое описание:
- Интерактивный веб-прототип игры в стиле Монополии, с элементами казино.
- Архитектура: Feature-Sliced Design (FSD)  `app`, `pages`, `processes`, `widgets`, `features`, `entities`, `shared`.
- Основные домены: игроки (`entities/player`), банк/казино (`entities/bank`), игровое поле (`entities/board`), ставки/раунды (`features/betting`).

Цели обновления знаний:
1. Принять изменения в кодовой базе и документации.
2. Обновить внутренние представления о новых сущностях, API и контрактах.
3. Сформировать краткий и структурированный changelog для разработчиков.

Формат входных данных (копировать в модель):
```
{
  "change_type": "file|feature|api|docs|config",
  "summary": "Короткое описание изменений (1 предложение)",
  "files_changed": ["путь/к/файлу1", "путь/к/файлу2"],
  "fsd_slice": "app|pages|features/...|entities/...|shared/...",
  "details": "Развернутое описание изменений и мотивов",
  "migration_steps": ["шаг 1", "шаг 2"],
  "version": "vX.Y.Z",
  "author": "Имя автора",
  "date": "YYYY-MM-DD"
}
```

Ожидаемый ответ от модели (шаблон):
1) Краткое резюме изменений (1-2 строки).
2) Обновлённая карта FSD (какие срезы затронуты).
3) Список файлов с кратким описанием изменений.
4) Рекомендации по интеграции / миграции (если нужны).
5) Обновлённый changelog-entry в формате:
```
- [version] YYYY-MM-DD  Автор: краткое описание (файлы: ...)
```

Вопросы, которые модель должна задать при неполных данных:
- Какие тесты покрывают изменения?
- Нужно ли обновлять интерфейсы API внешних сервисов?
- Есть ли обратная совместимость с текущими данными пользователей?

Пример использования (короткий):
1. Разработчик формирует JSON по формату выше и отправляет модели.
2. Модель возвращает структурированный changelog и список действий для CI.
